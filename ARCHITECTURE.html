Luly plugin — current architecture & server-side simplification path

Why install today is heavy, what's actually in the plugin, and how moving the analytical half to a hosted MCP server would let team members use the pipeline from claude.ai chat without Terminal / Node / Homebrew.

LLM-driven (skill markdown, just instructions) Analytical / scripted (Node + ts-node, runs locally) Mixed (LLM writes, script validates)

1. Where the install pain comes from

Today the plugin ships two completely different kinds of work bundled into one artifact, and both run on the user's machine:

  • Skills — markdown files that tell the LLM what to do. These are weightless: just text in plugins/luly/skills/<name>/SKILL.md. Anyone with Claude Code (or claude.ai with the Skills feature) could use them.
  • Scripts — TypeScript validators + the assembler in plugins/luly/scripts/*.ts. These need Node.js, ts-node, plus a few hundred MB of node_modules/ (TipTap, schema libs, etc.) to run.

Because the scripts run via ${CLAUDE_PLUGIN_ROOT}/bin/luly, every team member needs:

  1. Claude Code installed (Terminal / CLI version — the chat app doesn't support /plugin).
  2. Node.js on their machine.
  3. The plugin's node_modules/ (pulled in at install time).
  4. Successful execution permissions for the binary in the plugin cache.

For RG, AM, KF — this is more setup than it's worth. The skills (the markdown half) would work in much lighter surfaces; the scripts are the reason they can't.

2. Current pipeline — stage-by-stage

What each stage actually does, and which side runs it:

#StageWhat it producesLLM stepAnalytical step
0 luly-brief brief.json Interview user, capture intent / audience / tone, do brand research (WebFetch for company colors + logo) validate-brief.ts — schema check, HEX-color check on brand block
1 luly-product-type product-type.json Pick preset (academy / campaign-simple / waitlist / …), generate key, decide academy name vs course name validate-product-type.ts — preset enum, key format
2 luly-plan plan.md + plan.parsed.json Draft course outline as Markdown (lessons, screen synopses), iterate validate-plan.ts — regex-strict line parser, writes the parsed sidecar that all later stages consume
3 luly-format format-profile.json Decide format defaults (story vs responsive, quizzes on/off, locales) validate-format-profile.ts
4 luly-style theme.json Generate full 22-token palette + font pair, anchored to brand colors when present validate-theme.ts — token completeness + AA / AAA contrast checks
4b luly-icon card-cover.svg + course-icon.svg Design two SVG visuals (wide cover + square icon) using theme palette (planned: validate-icon.ts — SVG syntactic check, no script tags, viewBox match)
5 luly-fill-lesson lesson-N.json per lesson Write screens + blocks as JSON with Markdown content, image captions with HEX from theme validate-lesson.ts + block-validation.ts — block format enum, required field presence, quiz answer integrity, form field schema
5a luly-fill-onboarding onboarding.json Same as 5 but for onboarding screens validate-onboarding.ts
6 luly-overrides (optional) overrides.json Per-screen style overrides (rare) validate-overrides.ts
7 apply-controls controls.json (none — fully deterministic) apply-controls.ts + controls-presets.ts + validate-controls.ts — reads product-type + plan + lessons, computes per-node Back/Next/Skip/Continue controls, validates
8 luly-review-* (optional) Markdown review report Read all artifacts, flag drift / weak quizzes / brief misalignment (none)
9 assemble ⚡ <key>.luly.json (none) ⚡ The heavy one. assemble.ts + markdown-to-tiptap.ts + themes.ts + validate-flow.ts. Compiles every Markdown body to TipTap JSON, applies theme tokens, embeds controls, stamps slugs (uuidv4) + lexoRanks, inlines the SVGs, validates the final flow. ~80% of the node_modules/ footprint is here (TipTap pulls a lot).

Roughly: 60% of the work-by-character-count is LLM markdown (skills); 40% is local Node scripts. But 100% of the install pain is from the 40%.

4. The simplification — move analytical half to a hosted MCP server

Architecture sketch:

┌──────────────────────────────────────────┐         ┌──────────────────────────────┐
│   User's Claude (any surface):           │         │   Luly MCP server            │
│   - claude.ai web                        │  HTTPS  │   (hosted on Render /        │
│   - Claude Desktop chat app              │ ◄─────► │    Fly / Railway, ~$10/mo)   │
│   - Claude Code CLI                      │   MCP   │                              │
│   - Claude Code Desktop GUI              │         │   Exposes tools:             │
│                                          │         │   - luly_validate_brief      │
│   Skills (markdown) added via:           │         │   - luly_validate_plan       │
│   - claude.ai/customize/skills (chat),   │         │   - luly_apply_controls      │
│   - or local plugin (Claude Code only)   │         │   - luly_assemble  ⚡        │
└──────────────────────────────────────────┘         │   - … (one per validator)    │
                                                     └──────────────────────────────┘

What changes:

  • Skills stay on the user side — they're just instructions for the LLM. They can be delivered via the plugin (Claude Code users) OR uploaded to claude.ai/customize/skills (chat-app users). Same markdown either way.
  • Scripts move server-side — every ${CLAUDE_PLUGIN_ROOT}/bin/luly <script> call becomes an MCP tool invocation. The agent passes the artifact content as a tool param; the server runs the validator/assembler; the result comes back as JSON.
  • No local Node, no node_modules, no plugin binary install for non-technical users. They just add the Luly MCP connector (same UX as adding Gmail / Google Drive — which the team already does).
  • Artifacts move with the call — the assemble tool returns the final .luly.json as a string in the MCP response. The user saves it (or the agent does, via Claude's normal file-write tools) and uploads to the CMS.

7. Trade-offs

Pros

  • Removes the install barrier completely for RG / AM / KF — they use the pipeline in claude.ai chat with no Terminal, no Node, no plugin.
  • Updates ship server-side. No "uninstall v0-1-7, install v0-1-8" dance. Bug fix? Push to the server, every user has it next call.
  • Centralized telemetry / error logs. Today, when RG's generation fails, the error is on his machine. Server-side = AS can see what happened.
  • Stricter version control — the server is the single source of truth for validators + assembler. No drift between users.
  • Lighter plugin (if kept) — just skill markdown, no node_modules/, no bin/. Installs in seconds.
  • Opens the door to features that need server state (rate limiting, per-user history, shared templates) without forcing each user to host them.

Cons / risks

  • Adds a moving piece (server) that needs to be up. Render free tier sleeps after inactivity — first call after sleep takes ~30s to wake. Paid tier (~$7/mo) keeps it warm.
  • Network latency per call. The validators are tiny so this is fine; assemble is the heavy one but it's already slow enough that an extra 200ms doesn't matter.
  • Privacy: artifact content (brief, plan, lessons) crosses the network to the MCP server. Today it stays on the user's disk. Mitigations: HTTPS (assumed), no persistence on server side unless explicitly needed, document data flow.
  • Vendor lock-in to the MCP spec. If Anthropic changes the protocol, the server needs to keep up. Risk is low — MCP is stable and Anthropic is the one defining it — but worth naming.
  • One more thing for UM to maintain. Small ongoing cost.

9. Recommendation

Yes, do it. The investment is small (~3-4 dev-days, ~$10/mo) and removes the single biggest adoption blocker the team currently has. Order of operations:

  1. Stand up a minimal server with one tool (luly_validate_brief). Smoke-test that AS can call it from claude.ai chat via MCP.
  2. Port validators one at a time. Each one is tiny.
  3. Port assemble. This is the big one but it's pure code-move.
  4. Update skills to call MCP tools.
  5. Roll out to RG / AM / KF — they add the connector, follow a 3-line onboarding ("type /luly draft …"), use it.
  6. Decommission the local plugin path once everyone's on MCP (or keep it for AS/UM as a power-user option).
What this does NOT change: the LLM-driven half of the pipeline. Skill markdown stays exactly as it is. The agent still writes briefs, plans, lessons, themes by reading the same SKILL.md files. We're only moving the analytical 40%.

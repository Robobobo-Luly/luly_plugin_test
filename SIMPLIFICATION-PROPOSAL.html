Luly pipeline — simplification proposal (brainstorm v1)

For: AS. Goal: collapse 10-stage JSON-passing pipeline into ~5 stages of mostly-markdown context-passing. Keep all current functionality, drop the ceremony.

Keep Collapse / merge Drop entirely

1. The core insight (yours)

"Skills are just bunch of separate agents running on some context. The previous agents should just pass further the context. Why do we pass it in form of JSON and then check if the schema is valid — can't we just do the fucking md."

Yes. The JSON + validator-per-stage pattern was defensive engineering from when LLM output was less reliable. Today:

  • Inter-stage hand-offs can be natural-language markdown that the next agent reads and reasons over — no schema, no validator gate.
  • Validators only matter where they catch real downstream breakage (mostly: hex format on colors, final flow shape that the CMS importer expects).
  • The only stage that genuinely needs structured output is the final assemble, because the deliverable is a structured JSON the CMS imports.

3. The 5 new stages, in detail

Stage 1 — Intake

Skill: luly-intake (renames + absorbs luly-brief + luly-product-type).

What it does:

  • Reads the user's one-line prompt.
  • Pulls any URL the user attached via WebFetch for context.
  • If the prompt names a real company, does brand research (site, brand book, docs) to extract hex colors + logo URL. Standard inline.
  • Picks the preset (academy / academy-course / campaign-simple / waitlist / interactive-proposal / campaign-course) in the same step, with the same single-lesson rule and topic-noun trap rule as today.
  • Asks the user at most one clarifying question, only if a slot is genuinely ambiguous AND defaults won't work.
  • Writes a single intake.md: a human-readable summary that captures intent, audience, tone, materials, brand colors/logo/voice, chosen preset + rationale, generated key.

Output shape: markdown with semantic sections, e.g.:

# Intake — <key>

## What
<one-paragraph intent expansion>

## Audience
<sentence>

## Tone
<sentence>

## Preset
academy — academy-course / campaign-simple / waitlist / …
Rationale: <one line>

## Brand (when applicable)
Company: Phantom
Website: https://phantom.com
Colors:
- primary: #AB9FF2
- background: #FFFFFF
- accent: #4A4A4A
Logo: https://phantom.app/img/phantom-logo-purple.svg
Voice: friendly, confident, a little playful — ghost mascot energy

## Materials
- <url> — <one-line digest>
- <url> — <one-line digest>

No validator. Hex format errors are caught downstream in assemble.

Stage 2 — Plan

Skill: luly-plan (absorbs luly-format; format profile becomes a couple of lines in the plan doc).

Reads intake.md, writes plan.md. Same markdown grammar as today, plus a small frontmatter block for format toggles (story vs responsive, quizzes on/off, locales):

No separate plan.parsed.json. Assemble parses the markdown directly at stage 5.

One open question: should sections still be called ## Lesson N — title or something more generic? See §5 below.

Stage 4 — Fill

Skill: luly-fill (absorbs luly-fill-lesson + luly-fill-onboarding; one skill, one markdown artifact for ALL screens).

Reads intake + plan + theme. Writes a single content.md with all screens in order. Onboarding and lessons are both just "sections of screens" in the same doc. Quizzes / forms use fenced YAML-ish blocks where structure is needed; plain prose otherwise.

Why this is good:

  • One file holds the entire flow. Easy to inspect, easy for the agent to keep coherent voice across screens.
  • Markdown prose for narrative blocks (just write it).
  • Lightweight structured blocks for quiz/form schema (still readable, less brittle than JSON).
  • No per-lesson file fan-out.

Concern: long flows (8+ sections × 5 screens = 40+ screens) become a big file. Mitigation: the file can be sectioned and the agent can write it in passes (write sections 1-3, then 4-6, etc.) — but as a single artifact, not split files. Modern Claude handles 40 screens of content well within context.

Stage 5 — Assemble

Script: assemble.ts (only meaningful script remaining).

Reads all four markdown files + the two SVGs. Single pass:

  1. Parse intake.md → preset, key, brand, etc.
  2. Parse plan.md frontmatter + sections → tree shape.
  3. Parse theme.md → palette + fonts.
  4. Parse content.md → screens + blocks; compile each Markdown body to TipTap inline.
  5. Compute controls deterministically (same logic as today's apply-controls.ts, inlined here).
  6. Inline the two SVGs into course.body.cardImageSvg + course.body.iconSvg.
  7. Stamp slugs (uuidv4) + lexoRanks.
  8. Run a single final validator (the existing validate-flow.ts shape check).
  9. Write <key>.luly.json.

This is the only script that survives the cull. Everything else is markdown-driven by the agent.

5. Terminology — what to call "lesson"

You're right that "lesson" is wrong for waitlist / interactive-proposal / onboarding-only flows. They're all "ordered groups of screens."

Internally (data model + assembler): just call it section. Neutral, accurate, doesn't carry academy connotation.

User-facing label is derived from the preset:

PresetInternal typeUser-facing label in CMS
academy / academy-coursesection"Lesson"
campaign-coursesection"Step"
campaign-simplesection"Step" (when more than 1) or just unlabeled
waitlistsection(usually 1, no label)
interactive-proposalsection"Slide" or "Chapter"
onboarding screens (any preset)section with kind: onboarding(no label — they're pre-hub)

In agent-facing intake/plan/content markdown, also use ## Section N — title instead of ## Lesson N — title. Single uniform syntax across all presets. The orchestrator skill teaches the agent to write "Lesson" or "Step" in user-facing copy where appropriate.

Alternative names considered: "screen group" (clear but bureaucratic), "module" (academy-flavored), "chapter" (book-flavored), "flow" (ambiguous with the parent flow concept). I'd go with section internally. Open to other proposals.

6. What gets lost — honest list

  • Per-stage error feedback timing. Today, if the brief is malformed, you find out at stage 0. Tomorrow you'd find out at stage 5 (assemble). Mitigation: each skill self-checks in its prompt ("emit hex with #, 6 chars" — agent verifies before writing). Final assemble catches the rest. In practice modern Claude rarely produces structurally wrong markdown if the skill is clear.
  • Per-lesson granular validation. Today lesson 3's quiz integrity is validated independently. Tomorrow it's checked at assemble. If validation fails, the error points to "section 3 screen 2" in content.md — clear enough.
  • The parsed sidecar concept. plan.parsed.json went away. Assemble does its own parsing. One source of truth.
  • Per-stage validators as a teaching tool. Today a new dev can read each validator to learn the schema. Tomorrow they read assemble.ts and the skills. Slightly less granular but arguably clearer.

7. What stays the same

  • Six product presets — academy, academy-course, campaign-course, campaign-simple, waitlist, interactive-proposal. Unchanged.
  • Final .luly.json shape — unchanged. CMS importer doesn't need to know we restructured upstream.
  • Brand block (HEX colors + logo) — same fields, just lives in intake.md instead of brief.json.
  • Theme tokens — same 22 tokens, same font list, same contrast targets. Just stored as markdown.
  • Block formats — text, image-richtext, image, video, quiz-text, question, form-text, layout. Unchanged.
  • Controls computation — same logic (Back/Next, Skip/Continue for onboarding, etc.), just lives inside assemble.
  • SVG generation for card cover + course icon — unchanged.
  • The orchestrator UX (one-prompt → one JSON file) — unchanged from the user's POV.

8. Migration path

Rough order, ~2 days of work:

  1. Branch the plugin repo. Don't touch main until the new pipeline works end-to-end on a Phantom academy regenerate test.
  2. Write new skills first (luly-intake, luly-plan v2, luly-style v2, luly-fill). Cleaner ground-up.
  3. Update orchestrator to drive the 5 new stages.
  4. Refactor assemble.ts to read the markdown artifacts. The hard logic (Markdown→TipTap, theme apply, controls computation, slug stamping) stays; what changes is the input side. Bring in the parsers for intake/plan/theme/content from the old per-stage validators (they already parse this stuff — just consolidate).
  5. Retire the dropped scripts + skills. Move them to an archive/ folder for one release in case we need to reference them.
  6. Smoke test: regenerate Phantom academy. Check the final JSON byte-for-byte against the v0.1.7 output where it should match (same logic, same data). Differences should only be in things we explicitly changed.
  7. Bump to v0.2.0. This is a major structural change; the version reflects that.

9. Open questions for AS

  1. Terminology — "section" is my pick. Alternatives: chapter, module, screen-group. Strong opinion?
  2. Do we keep brand research separate or fold into intake? Current proposal folds it in. Pro: one stage. Con: intake is doing more work. I'd keep it folded — the agent does both reads in one turn anyway.
  3. The "review" agents — keep as optional standalone skills, or drop entirely? They're seldom used. I'd keep them as escape hatches.
  4. Single content.md vs one per section? Current proposal: one file. Concern flagged for very long flows. Alternative: one file per section but inside a content/ folder. Cleaner separation, same single-skill ownership.
  5. Markdown for theme vs JSON? Markdown is more inspectable; JSON is what assemble parses. I'd write theme.md as markdown, parse it into JSON in assemble. But theme.json kept directly is also reasonable. Mild preference for markdown for consistency.
  6. Backward compat with existing v0.1.x briefs? If a user has an old tmp/luly-agent/ dir from v0.1.7, do we want assemble to still understand it? I'd say no — v0.2 is a fresh start. Old artifacts get archived or deleted.

10. My recommendation

Go ahead. The biggest risk is the single content.md getting too long for very large academies — but in practice nobody's making 50-screen academies in one shot, and the agent can split the write into multiple passes against the same file. Everything else is upside.

Once approved, I'll implement on a branch and we ship as v0.2.0.
